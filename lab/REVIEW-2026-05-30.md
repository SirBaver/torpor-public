## F1 — CONFIRMED / HAUTE — Sentinel d'agrégat inaccessible à count == 101 + new-resource

**Fichier :** `poc/runtime/src/actor.rs:897`
**Statut :** [x]

**Problème.**
Dans `emit_cap_denied`, `cap_denied_count` est incrémenté *avant* le branchement. Si les 100 premiers événements passent par la branche scalaire (resources connues, branche 2), et que le 101e arrive sur une resource *neuve*, la branche 1 (`is_new_resource && has_room`) court-circuite et retourne `(true, false, 101)`. La branche 3 (`count == CAP_DENIED_RATE_LIMIT + 1`) n'est jamais atteinte. Le sentinel d'agrégat n'est pas émis dans cette fenêtre. Tout consommateur qui détecte la saturation du rate-limit via `is_aggregated = true` manquera le signal.

**Recommandation.**
Deux options :
- Option A : quand la branche 1 fire avec `count == RATE_LIMIT + 1`, émettre *deux* événements — l'événement attribué pour la resource neuve, puis le sentinel agrégé.
- Option B (plus propre) : introduire `aggregate_emitted: bool` dans `AgentState` (reset avec la fenêtre). Si `count > RATE_LIMIT && !aggregate_emitted`, forcer l'émission de l'agrégé sur le premier appel qui dépasse la borne, quel que soit le chemin (new-resource ou non).

## F2 — CONFIRMED / HAUTE — Invariant correctif #6 vacueux dès la 33e resource distincte

**Fichier :** `poc/runtime/src/actor.rs:895`
**Statut :** [x] — Option D (architect) : sentinel d'overflow de set (`cap_denied_set_overflow_emitted`) + correction docstring. HashMap rejeté (détruit l'invariant anti-DoS ADR-0051).

**Problème.**
Le commentaire ligne 879 garantit : *"une resource NOUVELLE force un événement attribué AVEC sa resource, même au-delà du rate-limit scalaire."* Cette garantie est vraie tant que `cap_denied_resources.len() < 32`. Dès que le BTreeSet est plein (`has_room = false`), la resource #33 — avec `count > 100` — tombe en branche 4 (silence). Aucun log, aucun diagnostic. La vulnérabilité SEF-9 (masquage d'un refus malveillant sous flood) est partiellement réouverte pour les scénarios avec plus de 32 resources distinctes.

**Recommandation.**
Remplacer le BTreeSet par un `HashMap<String, u32>` comptant les refus par resource. Le rate-limit scalaire s'applique *par resource* plutôt que globalement, ce qui supprime l'edge-case du set plein et rend la garantie du correctif #6 inconditionnelle. Si le volume de log reste une contrainte, la borne globale peut être maintenue comme `sum(counts) <= RATE_LIMIT` avec un cap par resource indépendant.

## F3 — CONFIRMED / MOYENNE — `get_header()` bloquant (RocksDB synchrone) dans `async fn restore_from_evicted`

**Fichier :** `poc/runtime/src/actor.rs:2118`
**Statut :** [x]

**Problème.**
`store_ref.get_header(&snap)?` est un appel synchrone (`fn get_header`, pas d'`await`) à l'intérieur d'une `async fn`. Il exécute `db.get_cf` — lecture RocksDB bloquante. Sous saturation I/O ou cache froid, il peut bloquer le thread exécuteur Tokio plusieurs millisecondes. Tous les tests et runners sef1/sef2/sef6 utilisent le flavor `current_thread` — un seul appel bloquant fige l'ensemble de l'exécuteur.

**Recommandation.**
Envelopper l'appel dans `tokio::task::block_in_place(|| store_ref.get_header(&snap))` (si le runtime est multi-thread) ou `tokio::task::spawn_blocking` (dans tous les cas). Alternativement, exposer `async fn has_snapshot(&self, id: &BlockHash) -> Result<bool, StoreError>` sur `ContentStore` qui utilise `spawn_blocking` en interne — ce qui résout aussi F7 (désérialisation inutile).

## F4 — PLAUSIBLE / MOYENNE — Fail-safe côté lecture, pas côté écriture

**Fichier :** `poc/runtime/src/actor.rs:2117`
**Statut :** [ ] *(#7b différé, lié au chantier GC — ADR-0049 §D3a)*

**Problème.**
Le correctif #7a détecte la référence pendante *au moment de la restauration de l'agent évincé*. Mais tout code lisant `last_snapshot` entre le crash et la prochaine restauration — `rollback_path`, lookup causal cross-agent, validation de chaîne de hash — voit la référence pendante sans déclencher la garde. La fenêtre reste ouverte.

**Recommandation.**
Ajouter une garde côté écriture dans le commit : après `put_snapshot` et avant `log_ref.append`, vérifier `get_header(snap_id).is_some()`. Si la vérification échoue, ne pas appender — erreur dure. Ce serait une prévention (fermeture de la fenêtre) plutôt qu'une détection tardive. Lié au chantier #7b (commit cross-store atomique).

## F5 — PLAUSIBLE / MOYENNE — Assertion A-E1 FIFO fondée sur `Instant` au lieu de `admission_seq`

**Fichier :** `poc/runtime/src/bin/p10_s5_runner.rs:283`
**Statut :** [x]

**Problème.**
La vérification d'ordre FIFO intra-classe compare `w[0].1 <= w[1].1` sur des valeurs `slot_acquired_instant: Option<Instant>`. Avec `pool_cap > 1`, deux agents admis dans l'ordre A < B par `admission_seq` peuvent acquérir les slots dans l'ordre B < A selon l'ordonnancement OS. La comparaison Instant produit alors un faux FAIL.

**Recommandation.**
Remplacer la comparaison `slot_acquired_instant` par `admission_seq` pour les assertions E1/E3. `admission_seq` est déjà disponible dans `QueueTrace` et reflète l'ordre d'admission logique, indépendant de l'ordonnancement OS.

## F6 — PLAUSIBLE / FAIBLE-MOYENNE — `wake_agent()` efface le type `MissingBlock` en `String`

**Fichier :** `poc/runtime/src/scheduler.rs:272`
**Statut :** [x]

**Problème.**
`wake_agent()` convertit toute `RuntimeError` en `DeliverError::WakeFailed(String)` via `map_err`. Après le correctif #7a, `restore_from_evicted` peut retourner `RuntimeError::Store(StoreError::MissingBlock(hash))`. L'information de type est perdue — tout futur appelant qui veut discriminer snapshot manquant vs erreur I/O doit parser la chaîne.

**Recommandation.**
Étendre `DeliverError` avec un variant `DanglingSnapshot(BlockHash)` et retourner ce variant explicitement quand `restore_from_evicted` renvoie `MissingBlock`. Cela préserve les garanties du système de types et permet une gestion différenciée (ex. quarantaine de l'état évincé vs retry).

## F8 — PLAUSIBLE / FAIBLE — `cap_denied_resources` utilise `BTreeSet` sans ordre trié observé

**Fichier :** `poc/runtime/src/actor.rs:646`
**Statut :** [x] *(résolu avec F10 : `CapDeniedLimiter` utilise `HashSet`)*

**Problème.**
Le set est accédé uniquement via `contains`, `insert`, `len`, `clear` — aucune opération n'exige un ordre trié. `BTreeSet` impose O(log n) par opération avec comparaison de chaînes ; `HashSet` donne O(1) amorti.

**Recommandation.**
Remplacer par `std::collections::HashSet<String>`. Si l'affichage ordonné dans les logs est souhaité, trier à ce seul moment-là. Note : ce finding devient caduc si F2 remplace le BTreeSet par un `HashMap<String, u32>`.

## F10 — PLAUSIBLE / FAIBLE — Cluster rate-limit sans type encapsulant

**Fichier :** `poc/runtime/src/actor.rs:637`
**Statut :** [x]

**Problème.**
`cap_denied_count`, `cap_denied_window_start_ms`, `cap_denied_resources` encodent un seul concept mais sont trois champs indépendants dans `AgentState`. Deux sites de reset doivent rester en cohérence manuellement (lignes 887–889 et 1212–1214). Un reset partiel laisserait des entrées périmées dans le set.

**Recommandation.**
Extraire un struct :
```rust
struct CapDeniedLimiter {
    count: u32,
    window_start_ms: u64,
    resources: HashSet<String>,
}
impl CapDeniedLimiter {
    fn reset(&mut self) { self.count = 0; self.window_start_ms = 0; self.resources.clear(); }
}
impl Default for CapDeniedLimiter { ... }
```
Un seul appel `.reset()` garantit la co-cohérence. Ce refactoring absorbe F8 et simplifie le corps de `emit_cap_denied`.
